关于v-if和v-show的踩坑记录

2019-09-26 14:00:51

本文用来记录v-if和v-show在弹框表单中的各种坑。

# 介绍

    最近在开发后台管理系统时，偶然踩到了一个自己平时不以为然的坑——关于v-if和v-show在form表单使用的问题。

# 正文

# 场景1

    我需要做一个弹框表单，于是使用了Element-UI的Dialog组件嵌套Form组件，（表单需要校验非空），到这里是没有问题的。但当我自己测试时发现先点新增表单然后出发非空校验提示，之后关闭表单再打开编辑表单时就会出问题————上一次的校验结果没有清空，导致本来有值的项还是会提示为空。

    对于上面的问题的解决思路很简单————每次都使用resetFields()校验结果的进行清空即可。那么问题来了，清空操作应该放在关闭上一个表单时还是新开表单时？
我的第一想法是后者，但是后来发现新开表单时dom并不能及时挂载（即使设置了this.$nextTick）后来看到有文章说是因为Dialog和Form两个dom挂载的时间不一致导致的，因此当Dialog打开时触发resetFields()，但使用的是Form的方法（此时Form为挂载，因此通过ref获取不到），所以会出现报错。

    报错的原因就是因为打开弹框的时候获取不到dom节点，当然这时可能会想到this.$nextTick（然而并无作用。。。），其实有种懒一点的解决方案就是设置定时器（这里我试了下0.5s就可以），但是随着表单的增加这种方法毕竟不牢靠。

    对于这种问题的解决方案：在表单关闭时触发close事件(而不是打开表单的open事件)清空校验结果，其原理就是因为当打开表单时dom没有挂载完成，因此会出问题；但关闭的时候清空校验结果，这时dom还没有卸载，因此可以先清空校验结果。

# 场景2

    接上面的例子，如果我想设置一个复杂的表单（item多于10），并且有些项不想被编辑（比如说新增时有10项，而编辑时只显示8项）。而这时我们会面对着使用v-if还是v-show来控制显隐。

    上图是使用v-if来进行控制时报的错（当然这个问题并不常见，一般需要比较复杂的dom结构），使用v-if来控制时，如果先点击编辑（显示8项），再点击新增（显示10项）关闭时则会报错。

    原理分析：当表单结构比较复杂时，使用v-if来控制显隐会出现dom增删操作频繁，此时获取dom就很有可能获取不到。

# 总结

引用官网解释：

v-if 是“真正”的条件渲染，因为它会确保在切换过程中条件块内的事件监听器和子组件适当地被销毁和重建。

v-if 也是惰性的：如果在初始渲染时条件为假，则什么也不做——直到条件第一次变为真时，才会开始渲染条件块。

相比之下，v-show 就简单得多——不管初始条件是什么，元素总是会被渲染，并且只是简单地基于 CSS 进行切换。

一般来说，v-if 有更高的切换开销，而 v-show 有更高的初始渲染开销。因此，如果需要非常频繁地切换，则使用 v-show 较好；如果在运行时条件很少改变，则使用 v-if 较好

    对于v-if和v-show，之前我的理解仅仅是以为v-show是css层面的，v-if是html层面的（虽然就是这样）。出于懒惰一般惯用v-if，但是经过复杂项目的洗礼后才知道原来v-if也不是随便用的。

# 结论

    对于可能频繁切换或涉及到复杂dom时可以采用v-show来控制显隐开销；而对于切换不太频繁或者像是移动端展示最好还是使用v-if来节省性能。
